Product Story

A PM's experiment
with AI-assisted building

How a product manager with no engineering team shipped a real, multi-user product in six weeks — and what that revealed about the future of product work.

01 — Origin

How it came into existence

Every product starts with a problem that's too personal to ignore.

"One person was
carrying the mental
load. The other
didn't even know
there was a load."

It started with a Sunday evening conversation — the kind that couples have when one person has been mentally tracking fifteen things all week and the other doesn't even know there's a list. "Did you remember to call the landlord?" "I thought you were doing that." "No, I told you in the car."

Tasks were scattered across WhatsApp threads, Apple Reminders, and two heads that weren't always in sync. Ideas would come up at dinner and be forgotten by morning. Worries got carried silently. The mental load was split unevenly — not out of carelessness, but out of a lack of shared infrastructure.

The tools available were either too simple (WhatsApp) or too complex (Notion, Jira). Nobody wants to set up a project management workflow for "did we order coffee pods?"

As a product manager, I knew exactly what I wanted: something that felt like texting but organised everything automatically. A shared brain for the household — always in sync, always visible, zero friction to add things.

Instead of waiting for a team to build it, I opened Claude AI and started describing the product I wished existed. Six weeks later, Second Brain was live and being used daily.

03 — Process

How it was built

Every feature was a product decision first, a technical conversation second. Here's the full journey — in order.

1
Discovery

Defined the problem clearly

Mapped out exactly what was broken: tasks lost in WhatsApp, no shared visibility, unequal mental load, no easy capture mechanism. Wrote a one-page problem statement before touching any technology.

2
Research

Audited existing tools and identified the gap

Todoist — too personal. Notion — too complex. Alexa — voice only, no board. WhatsApp — no organisation. The gap was clear: nobody had built a shared kanban that felt as easy as texting.

3
Tech Stack

Chose the right stack for the job

React + TypeScript + Vite for the app (component structure, type safety, fast builds), Firebase Realtime Database (real-time sync out of the box), Firebase Auth (Google Sign-In in three lines), GitHub Pages + GitHub Actions (free, auto-deploys on push to main). Landing and story pages stay as plain HTML — no build step needed there.

4
MVP

Built the kanban board with a chat-based input

No forms. No dropdowns. Just a text box where you type anything naturally and the board fills up. Eight columns — Tasks, Worries, Ideas, Purchases, Trips, Life Admin, Other, and Completed.

5
Auth

Added Google Sign-In and multi-household architecture

Any Google account can sign in. Each household gets a private board with a 6-letter invite code. Firebase security rules ensure complete data isolation between households.

6
AI / NLP

Built Claude-powered categorisation with a keyword fallback

Every message is sent to a Cloudflare Worker that calls the Claude API (Haiku). Claude receives the full board state and conversation history, decides the intent (add / query / chat), picks a category, and writes a warm reply. If the network is unavailable, the app falls back to score-based keyword matching so it always works.

7
Mobile UX

Redesigned the layout for phones

Desktop shows all columns side-by-side. On mobile, columns stack and scroll vertically so the board fits any screen size. Action buttons (complete, delete, reassign) are always visible — no hover required.

8
Card Features

Added due dates and notes to every card

Click any card to open a detail view: set a due date (overdue cards turn red automatically) and add free-form notes or links. Everything saves in real time to Firebase.

9
Sharing

Built the invite system with email and direct links

Invite modal shows a 6-letter code plus a shareable URL (?code=XXXXXX) that pre-fills the join form. Email invite opens the sender's email app with everything pre-written — recipient just clicks the link.

10
Analytics

Added household insights and a weekly digest

Insights modal shows pending, completed, and overdue counts, active items by category, and contributions per member. Weekly digest summarises everything added in the past 7 days — useful for a quick household catch-up.

11
Distribution

Built a landing page, demo mode, and this story page

Landing page with live stats (boards created, items captured, members joined) pulled from Firebase. Demo mode lets anyone try a pre-populated board with no sign-in. This page tells the PM story behind the product.

04 — Decisions

The tradeoffs

Every product decision is a tradeoff. Here's what was chosen and why.

Decision Choice made Why
Decision
Single HTML file vs React / Next.js
Chose
React + TypeScript + Vite
Why
Started as a single HTML file for speed — zero tooling, one git push. Migrated to React + Vite as the feature set grew and component complexity justified a proper build step. Landing page and story remain plain HTML for simplicity.
Decision
Firebase vs custom backend
Chose
Firebase
Why
Real-time sync, auth, and database in three lines of config. No server to manage, no ops burden. The free tier handles thousands of users. Custom backend would have cost weeks and added zero user value.
Decision
Claude API for NLP vs keyword matching
Chose
Keywords (for now)
Why
Started with pure keyword matching — no API cost, works offline, handles 90%+ of inputs accurately. Claude API (Haiku) now powers categorisation in production via a Cloudflare Worker proxy, with keyword matching as the offline fallback. Best of both: always works, smarter when online.
Decision
Private tool vs open product
Chose
Open product
Why
Invite codes mean each household is completely isolated — opening it up costs nothing in security. But it multiplies potential users, enables a real demo, and turns this into a genuine PM portfolio piece with real traction.
Decision
Thin MVP vs feature-rich launch
Chose
Feature-rich
Why
A thin slice wouldn't demonstrate product thinking. The goal is to show what AI-enabled PM execution looks like at its best — shipping real features (voice, analytics, invite flows, due dates) that real users actually need.
Decision
category = flag vs moving cards on complete
Chose
completed: true flag
Why
Preserving the original category means the digest and analytics still know what type of thing was completed. If you mark a Task done, it still counts as a Task in your weekly summary — not lost to a generic "Done" bucket.
05 — The Experiment

What building with AI taught me about product management

I've worked as a PM long enough to know what the traditional process feels like. This was different in ways I didn't expect — and in some ways, harder.

Traditional PM
  • 📝Write a spec. Hope the engineers interpret it the way you meant.
  • ⏳Wait days or weeks for a build. Discover the gap between vision and reality at sprint review.
  • 🔄Iterate through formal sprint cycles. Each change takes a ticket, a standup, a PR review.
  • 🧑‍💻Lean on engineering to translate product instincts into working software.
  • 📊Measure success months after launch when enough usage data exists.
AI-Assisted PM
  • 💬Describe the experience you want. The AI drafts it. You review, react, refine — in the same hour.
  • ⚡Ideas go from thought to deployed feature in hours. The feedback loop is measured in minutes, not weeks.
  • 🔁Iterate freely. The cost of trying something is nearly zero — so you try more, discard faster, ship better.
  • 🧠You ARE the engineer. You have to read, review, and own every line — including the ones that look right but aren't.
  • 🚀Ship to real users immediately. Real feedback on day one, not month six.
⚡

The feedback loop compressed from weeks to hours

In a traditional PM role, you write a spec, wait for a sprint, and see the result two weeks later. With AI, I could describe a feature, see it working, use it, break it, and redesign it — all in an afternoon. That speed changes how you think about product decisions. You stop over-planning and start over-testing.

🧭

Your product instincts matter more, not less

AI can generate code, suggest architecture, and propose UX patterns — but it cannot feel the friction. It cannot tell you that a feature is technically correct but emotionally wrong. The PM skill of knowing what users actually need, and translating that into precise language, became the most valuable thing I brought to every session.

🎯

Prompt quality is the new requirements doc

The clearer and more specific your prompt, the better the output. Vague specs produce vague code — same as they always did with engineering teams, just faster. The discipline of writing a crisp, unambiguous description of what you want turned out to be exactly the skill that separates good PMs from great ones.

Pitfalls I ran into
🌀
AI hallucinations — confident, wrong, and convincing

The AI would occasionally generate code that looked completely correct but contained subtle bugs — wrong Firebase paths, non-existent SDK methods, logic that broke edge cases. The danger isn't that AI makes obvious mistakes. It's that it makes plausible mistakes. You have to review everything with the same scrutiny you'd apply to a junior engineer's first PR. Trust but verify — always.

📉
Context drift over long sessions

As conversations grew long, the AI would start losing track of earlier decisions — suggesting approaches that contradicted things we'd already resolved, or forgetting the architecture we'd agreed on. I learned to keep a running CLAUDE.md file (a context document for the AI) that summarised the project state, key decisions, and constraints. Without it, sessions in week 5 felt like starting over.

⚠️
Fast iteration can outrun your judgment

When you can ship in hours, there's a temptation to keep shipping. I added features I later removed. I rebuilt the same flow three times before realising the original was better. Speed is a multiplier — it multiplies good decisions and bad ones equally. The PM discipline of slowing down to ask "should we?" before "can we?" matters even more when the answer to "can we?" is almost always yes.

🔧
The AI rewrote the architecture mid-project

Partway through, the AI suggested migrating from a single HTML file to React + TypeScript + Vite — a legitimate upgrade, but it temporarily broke the deployed app and required re-implementing features I'd already shipped. AI will propose the "right" technical choice without fully accounting for the cost of migration. As the PM, you have to weigh that cost yourself. AI optimises for clean code; you have to optimise for the whole product.

06 — Why it's different

What no other app does

The household management category has plenty of tools. What's missing is the right combination — built around how households actually think and communicate, not how project teams work.

The three pain points it solves
🧠

The mental load is invisible and asymmetric

One person is mentally tracking everything — appointments, renewals, things said in passing. The other person isn't lazy; they just don't have visibility. No existing app addresses this as its primary design goal. They all assume both people are already engaged.

✍️

Every other tool makes you decide before you add

Open Todoist → choose project → type. Open Notion → navigate → type. WhatsApp → send → watch it disappear in three days of messages. Second Brain: just type anything naturally — it figures out where it goes.

💭

Emotional items have no home anywhere

Every task manager is built around action items. But a huge part of household mental load is emotional — worries, anxieties, "I've been thinking about this and I don't know what to do with it." No mainstream app acknowledges this exists.

The combination no one else has built
01
Shared-first, not personal-first

Designed from the start for two or more people with equal visibility. Not a personal app with a "share" button tacked on.

02
Natural language → auto-categorisation

Type anything naturally. Don't decide where it goes — the app decides. Closer to talking to someone than filling in a form.

03
Kanban visual output

Everything organises into a shared board you can both scan in 5 seconds. Not a list, not a calendar — a board that shows the whole picture.

04
Categories that match real life

Tasks, Worries, Ideas, Purchases, Trips, Life Admin. Not "To Do / Doing / Done." Categories that map to how households actually think.

05
Real-time sync, zero setup

Sign in with Google, share a link. No app to download, no workspace to configure. Everything syncs the moment it's typed.

💭

The Worries column — genuinely novel

Every task manager in existence is built around action items. But a huge part of household mental load is emotional — worries, anxieties, things you're carrying but don't know what to do with yet. There is no mainstream app with a Worries column. That category alone is genuinely novel. It validates emotional load as a real thing worth sharing with a partner — not something to suppress silently until it becomes an argument.

"WhatsApp has the friction right but the organisation wrong. Todoist has the organisation right but the household wrong. Notion has the power right but the simplicity wrong. Second Brain is the first tool designed around how households actually think — not how project teams work."
